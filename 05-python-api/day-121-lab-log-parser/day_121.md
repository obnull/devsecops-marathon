# Stage 01: Python & API
## Day 121: [LAB] Log Parser with HTTP Status Counter

## Objective
Build a complete log analysis tool from scratch, integrating all Week 17 skills.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Python: 3.12
- Tools: venv (shared from Day 116)

## Description
Wrote standalone log parser reading access.log, parsing into structured data, calculating statistics (total requests, unique IPs, status distribution, top IPs, 404 URLs), generating formatted report saved to file.

## What Was Done
- Designed 6 functions: parse_line, load_logs, count_statuses, top_ips, find_404_paths, generate_report
- Parsed logs by splitting and extracting ip, method, path, status
- Counted status codes by category (2xx/3xx/4xx/5xx)
- Found top 3 IPs via dictionary counting and sorted()
- Collected unique 404 paths with deduplication
- Generated formatted report and saved to report.txt

## Commands Used
- `python3 log_parser.py`
- `cat report.txt`

## Verification
- Total requests: 14
- Unique IPs: 4
- Status distribution correct
- Top IPs identified
- 404 URLs listed
- Report saved

## Issues Encountered
1. **Status counts showed zeros** — parse_line used wrong indices due to double spaces in log format. Fixed by adjusting indices from [5,6,8] to [4,5,7].
2. **File not found** — running from wrong directory. Fixed by cd into src/.
3. **find_404_paths logic** — second if not nested. Fixed by indenting.
4. **File write mode** — open() without 'w'. Fixed by adding 'w'.
5. **Method field had path** — strip('"') on wrong index. Fixed with index correction.

## Key Concept
Structured log parsing is observability foundation. ELK, Splunk, Loki do what this script does at scale. Understanding parse-extract-aggregate-report logic is essential for debugging monitoring tools.

## Takeaways
1. Debug by printing intermediate values
2. split() collapses multiple spaces, shifting indices
3. Nested ifs require careful indentation
4. open() defaults to read mode
