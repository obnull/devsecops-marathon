# Stage 01: Python & API
## Day 133: [LAB] Web Service Health-Checker with Alerts

## Objective
Build a real DevOps monitoring tool that checks availability of multiple web services and reports their status.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Python: 3.12
- Tools: venv, VSCode, python-dotenv, requests

## Description
Built a web service health-checker that reads endpoints from .env, makes HTTP requests with timeout, and generates a status report. Handles three failure scenarios: non-200 status, connection error, and read timeout. Simulates a real uptime monitoring script.

## What Was Done
- Stored endpoints as comma-separated string in .env
- Parsed ENDPOINTS string with `.split(",")` into a list
- Wrote `check_endpoint()` with timeout=5 and three exception handlers
- Caught `ConnectionError` for unreachable hosts
- Caught `Timeout` for slow/hanging services
- Wrote `main()` iterating over all endpoints and printing report
- Tested against google.com (200), github.com (200), httpbin.org/status/503, nonexistent.xyz

## Commands Used
- `python3 main.py`

## Verification
- google.com and github.com return 200 and log INFO ✅
- httpbin.org/status/503 returns 503 and logs WARNING ✅
- nonexistent.xyz logs ERROR and returns None status ✅
- ReadTimeout caught separately from ConnectionError ✅

## Issues Encountered
1. **ReadTimeout not caught** — `httpbin.org` timed out but script crashed because only `ConnectionError` was caught. Fixed by adding `except requests.exceptions.Timeout`.
2. **Intermittent DNS failure** — github.com showed as ALERT mid-run due to WSL DNS reset. Fixed with `echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf`.

## Key Concepts
- `timeout=5` in requests.get() prevents infinite hang on slow services
- `requests.exceptions.Timeout` catches both ConnectTimeout and ReadTimeout
- `requests.exceptions.ConnectionError` catches DNS failures and refused connections
- Comma-separated env vars parsed with `.split(",")` — standard pattern for list configs

## Takeaways
1. Always set timeout in requests.get() — never leave it open-ended in production
2. Handle Timeout and ConnectionError separately — different failure modes
3. Store list configs in .env as comma-separated strings, parse in code
4. Health checkers are the simplest form of production monitoring — build them first
