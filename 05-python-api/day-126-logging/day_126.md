## Day 126: Logging Instead of print()

## Objective
Replace print() statements with structured logging using Python's logging module.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Python: 3.12
- Tools: venv, VSCode

## Description
Upgraded the Day 125 config reader to use structured logging instead of print(). Logs written to file with timestamps and severity levels. Each function logs its own outcome at the appropriate level.

## What Was Done
- Added `logging.basicConfig()` with level, format, and filename
- Replaced `print()` with `logging.error()` for error cases
- Added `logging.info()` on successful file load
- Added `logging.info()` for parsed config output in __main__
- Verified log output written to `app.log`

## Commands Used
- `python3 main.py`
- `cat app.log`

## Verification
- Successful read logs INFO with filename ✅
- Missing file logs ERROR with path ✅
- Parsed config logged with full dict ✅
- app.log created in script directory ✅

## Issues Encountered
1. **app.log saved to wrong directory** — ran script from repo root, log created there instead of day folder. Fixed by running from correct directory with `cd` first.
2. **logging.INFO vs logging.info** — used uppercase method name which doesn't exist. Fixed to lowercase `logging.info()`.

## Key Concepts
- `logging.basicConfig()` configures the root logger once at startup
- Log levels by severity: DEBUG → INFO → WARNING → ERROR → CRITICAL
- `filename=` in basicConfig redirects logs from terminal to file
- Format string `%(asctime)s - %(levelname)s - %(message)s` adds timestamp and level automatically

## Takeaways
1. Never use print() in production scripts — use logging
2. Match log level to severity: INFO for normal flow, ERROR for caught exceptions
3. Always run scripts from their own directory or use absolute paths for log files
4. logging methods are lowercase: logging.info(), not logging.INFO()
