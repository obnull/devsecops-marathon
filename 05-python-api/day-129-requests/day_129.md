# Stage 01: Python & API
## Day 129: requests Module — HTTP, Headers, Status Codes

## Objective
Learn to make HTTP requests from Python, handle response codes, and catch connection errors.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Python: 3.12
- Tools: venv, VSCode

## Description
Built an HTTP service checker using the requests library. Script makes GET requests, evaluates status codes, logs results at appropriate levels, and handles unreachable hosts gracefully without crashing.

## What Was Done
- Wrote `check_service()` with `requests.get()` and status code evaluation
- Logged INFO for 200, WARNING for any other status
- Caught `requests.exceptions.ConnectionError` for unreachable hosts
- Tested against google.com (200), httpbin.org (503), and nonexistent.xyz (connection error)
- Observed automatic redirect handling — 301 → 200

## Commands Used
- `python3 main.py`
- `pip install requests`

## Verification
- google.com returns 200 and logs INFO ✅
- httpbin.org returns 503 and logs WARNING ✅
- nonexistent.xyz logs ERROR and returns None ✅
- Redirect from google.com followed automatically ✅

## Issues Encountered
1. **Missing f-string prefix** — wrote `logging.warning("WARNING {url}")` without `f`. Variable not substituted. Fixed by adding `f` prefix.
2. **WSL DNS failure** — `api.github.com` unreachable due to DNS reset. Fixed with `echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf`.

## Key Concepts
- `requests.get(url)` — sends HTTP GET request
- `response.status_code` — numeric HTTP response code
- `requests.exceptions.ConnectionError` — raised when host is unreachable
- requests follows redirects automatically by default

## Takeaways
1. Always wrap requests calls in try/except for ConnectionError
2. Check status_code, not the response object itself
3. WSL DNS can reset after reboot — fix with nameserver 8.8.8.8
4. requests handles redirects silently — final status_code reflects the last response
