# Stage 01: Python & API
## Day 128: os Module, subprocess — Running Linux Commands

## Objective
Learn to execute Linux system commands from Python using the subprocess module.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Python: 3.12
- Tools: venv, VSCode

## Description
Built a system resource checker that runs Linux commands from Python. Uses subprocess.run() to execute df and free, captures output and logs results. Demonstrates how DevOps scripts replace bash one-liners with structured Python tooling.

## What Was Done
- Wrote `check_disk()` using `subprocess.run(["df", "-h", path])`
- Wrote `check_memory()` using `subprocess.run(["free", "-h"])`
- Used `capture_output=True, text=True` to get stdout as string
- Logged completion of each check with INFO level
- Printed results in __main__ block

## Commands Used
- `python3 main.py`

## Verification
- Disk usage for "/" returned and printed correctly ✅
- Memory usage returned and printed correctly ✅
- Both functions log INFO on completion ✅

## Issues Encountered
1. **capture_output inside command list** — placed `capture_output=True` inside the list `["free", "-h", capture_output=True]`. Fixed by moving it outside as a separate argument.
2. **check_memory() called twice** — called once to save result and again inside print(). Fixed by saving return value and printing the variable.

## Key Concepts
- `subprocess.run()` executes a Linux command from Python
- Command passed as list — each word is a separate element
- `capture_output=True` — captures stdout and stderr into result object
- `text=True` — returns output as string instead of bytes
- `result.returncode` — 0 means success, non-zero means error

## Takeaways
1. Always pass commands as lists, never as a single string
2. capture_output and text are separate arguments, not part of the command list
3. subprocess.run() is the modern replacement for os.system()
4. Save return values to variables — avoid calling functions twice
