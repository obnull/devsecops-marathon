# Stage 01: Python & API
## Day 132: Environment Variables & .env (python-dotenv)

## Objective
Learn to manage secrets and configuration using environment variables and .env files instead of hardcoding values in source code.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Python: 3.12
- Tools: venv, VSCode, python-dotenv

## Description
Built a secrets loader that reads sensitive configuration from a .env file using python-dotenv. Validates that all required variables are present and logs WARNING for any missing ones. Token never appears in source code.

## What Was Done
- Created `.env` file with GITHUB_TOKEN, MONITOR_INTERVAL, TARGET_HOST
- Added `.env` to `.gitignore` to prevent accidental commit
- Installed `python-dotenv` with pip
- Wrote validation loop checking all required variables after `load_dotenv()`
- Loaded each variable with `os.getenv()` and printed values

## Commands Used
- `pip install python-dotenv`
- `echo ".env" >> .gitignore`
- `python3 main.py`

## Verification
- All three variables loaded correctly ✅
- Missing variable triggers WARNING log ✅
- .env excluded from git tracking ✅

## Issues Encountered
1. **Validation before load_dotenv()** — checked required vars before loading .env, all showed as missing. Fixed by moving `load_dotenv()` before the validation loop.

## Key Concepts
- `.env` file stores secrets as `KEY=value` pairs — no quotes, no spaces around `=`
- `load_dotenv()` loads .env into environment variables at runtime
- `os.getenv()` returns None if variable missing — safe alternative to `os.environ[]`
- `.env` must always be in `.gitignore` — never commit secrets

## Takeaways
1. Never hardcode tokens, passwords, or URLs in source code
2. load_dotenv() must be called before any os.getenv() calls
3. Always validate required variables at startup — fail fast with clear message
4. .gitignore is the last line of defence before secrets leak to GitHub
