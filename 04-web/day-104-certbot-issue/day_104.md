# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 104: Certbot & Let's Encrypt (mkcert)

## Objective
Obtain a trusted SSL certificate for local development using mkcert. Understand how Let's Encrypt and the ACME protocol work.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Tools: mkcert, curl, openssl

## Description
Installed mkcert, created a local Certificate Authority, and generated a trusted certificate for `proxy.local` and `www.proxy.local`. Configured Nginx to use the certificate. Browser shows a green padlock without warnings.

## What Was Done
- Installed mkcert and added it to the system trust store (`mkcert -install`)
- Generated certificates: `proxy.local.pem` and `proxy.local-key.pem`
- Placed certificates in `~/devsecops-marathon/04-web/certs/`
- Updated Nginx config with `ssl_certificate` and `ssl_certificate_key`
- Fixed path mismatch: certificates were in `certs/`, config pointed to `04-web/certs/`
- Fixed `curl: unable to get local issuer certificate` by re-running `mkcert -install`

## Commands Used
- `sudo apt install libnss3-tools -y`
- `curl -JLO "https://dl.filippo.io/mkcert/latest?for=linux/amd64"`
- `chmod +x mkcert-v*-linux-amd64`
- `sudo mv mkcert-v*-linux-amd64 /usr/local/bin/mkcert`
- `mkcert -install`
- `mkcert -cert-file proxy.local.pem -key-file proxy.local-key.pem proxy.local www.proxy.local`
- `sudo nginx -t && sudo systemctl reload nginx`
- `curl https://proxy.local:8443/`
- `mkcert -CAROOT`

## Verification
- `https://proxy.local:8443/` opened in browser with a green padlock
- `curl https://proxy.local:8443/` worked without the `-k` flag (after `mkcert -install`)
- `sudo nginx -t` — syntax ok
- `ls ~/devsecops-marathon/04-web/certs/` showed both `.pem` files

## Issues Encountered
1. **Certificates not found** — config path pointed to `04-web/certs/`, but files were in `~/devsecops-marathon/certs/`. Fixed by copying files.
2. **curl: unable to get local issuer certificate** — curl did not see the root CA. Fixed by running `mkcert -install`.
3. **Unexpected "}" in config** — stray backslash before `;`. Fixed by removing the backslash.

## Key Concept
**mkcert vs Let's Encrypt.** Let's Encrypt requires a real domain and a server reachable from the internet. mkcert creates a local CA trusted only by your machine. For development and testing, mkcert is ideal. In production, only Let's Encrypt or paid certificates should be used. The ACME protocol (used by Let's Encrypt) verifies domain ownership via a challenge file placed on the web server.

## Takeaways
1. mkcert is a fast way to get a trusted certificate for localhost.
2. A certificate = `.pem` file (public part) + key `.pem` file (private part).
3. Nginx requires both: `ssl_certificate` and `ssl_certificate_key`.
4. Always verify file paths in the config after generating certificates.
5. `mkcert -install` needs to run once — it adds the root CA to the system.
