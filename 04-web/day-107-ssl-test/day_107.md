# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 107: SSL Security Testing

## Objective
Audit HTTPS server security using testssl.sh and nmap. Identify weak ciphers, outdated protocols, and known vulnerabilities.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Tools: testssl.sh, nmap, openssl

## Description
Installed and ran testssl.sh against the local HTTPS server. Analyzed the report: confirmed TLS 1.0/1.1 are rejected, strong ciphers are used, and no known vulnerabilities exist. Identified areas for improvement: CBC ciphers, chain of trust, OCSP stapling. Removed remaining CBC ciphers from the config to pass the LUCKY13 check.

## What Was Done
- Cloned testssl.sh from GitHub
- Ran full audit: `./testssl.sh https://proxy.local:8443`
- Reviewed protocol support: TLS 1.2 and 1.3 only, SSLv2/SSLv3/TLS 1.0/TLS 1.1 not offered
- Reviewed cipher categories: NULL, Anonymous, Export, LOW, RC4 — all not offered
- Reviewed vulnerabilities: Heartbleed, POODLE, BEAST, FREAK, DROWN, LOGJAM — all not vulnerable
- Identified LUCKY13 as potentially vulnerable due to CBC ciphers
- Removed CBC ciphers from `ssl_ciphers` directive
- Re-ran audit: LUCKY13 resolved
- Ran nmap SSL cipher enumeration as alternative check

## Commands Used
- `git clone --depth 1 https://github.com/drwetter/testssl.sh.git`
- `./testssl.sh https://proxy.local:8443`
- `./testssl.sh --protocols https://proxy.local:8443`
- `./testssl.sh --ciphers https://proxy.local:8443`
- `sudo apt install nmap -y`
- `nmap --script ssl-enum-ciphers -p 8443 proxy.local`
- `sudo sed -i 's|ssl_ciphers .*|ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;|' /etc/nginx/sites-available/proxy.local`
- `sudo nginx -t && sudo systemctl reload nginx`

## Verification
- testssl.sh confirmed: TLS 1.2 and 1.3 offered, SSLv2/SSLv3/TLS 1.0/TLS 1.1 not offered
- All cipher categories except CBC: not offered (OK)
- Forward Secrecy: offered (OK)
- All vulnerabilities: not vulnerable (OK)
- After CBC removal: LUCKY13 resolved
- Client simulations: all modern browsers connect with TLS 1.3 + AES-256
- nmap showed only strong ciphers with grade A

## Issues Encountered
1. **LUCKY13 potentially vulnerable** — caused by CBC ciphers in the cipher list. Fixed by removing all CBC ciphers, keeping only GCM and ChaCha20.
2. **Chain of trust incomplete** — expected for mkcert, would be resolved with Let's Encrypt in production.
3. **502 Bad Gateway on `/`** — Flask was not running during the scan. HTTPS itself was fine.

## Key Concept
**SSL security auditing.** HTTPS configuration must be audited regularly. Tools like testssl.sh check protocols, ciphers, and known vulnerabilities (Heartbleed, POODLE, BEAST, etc.). The goal is an A+ rating on SSL Labs (for public sites) or a clean testssl.sh report (for internal sites). Key indicators: no SSLv2/SSLv3, no TLS 1.0/1.1, no NULL/EXPORT/LOW/RC4 ciphers, no CBC ciphers, Forward Secrecy enabled, no known vulnerabilities. OCSP stapling and HSTS are needed for the highest rating.

## Takeaways
1. testssl.sh is the go-to tool for local SSL auditing.
2. A green padlock does not mean the server is secure — ciphers and protocols must be checked.
3. CBC ciphers are obsolete and vulnerable to LUCKY13 — use only GCM and ChaCha20.
4. nmap `ssl-enum-ciphers` script is a good alternative for cipher enumeration.
5. Regular audits catch misconfigurations before attackers do.
