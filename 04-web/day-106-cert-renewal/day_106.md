## Day 106: Automated Certificate Renewal

## Objective
Set up automated SSL certificate expiry checking and renewal using cron and systemd timer. Understand certificate lifecycle management.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Certificates: mkcert
- Tools: cron, systemd, bash

## Description
Created scripts for checking certificate expiry and renewing certificates. Configured cron for weekly expiry checks. Set up a systemd timer as a modern alternative. Learned how certificate renewal works with Certbot and Let's Encrypt in production.

## What Was Done
- Checked current certificate expiry date with `openssl s_client`
- Wrote `ssl-check.sh` — script that checks remaining certificate validity days
- Wrote `ssl-renew.sh` — script that regenerates mkcert certificate and reloads Nginx
- Configured cron job: runs `ssl-check.sh` every Sunday at 3:00 AM
- Created systemd service and timer for weekly certificate checks
- Enabled and started the systemd timer
- Verified cron and systemd timer status

## Key Concept
**Certificate lifecycle management.** Certificates have an expiry date. Let's Encrypt certificates are valid for 90 days, mkcert certificates for 2+ years. In production, renewal must be automated: Certbot creates a cron job or systemd timer that runs `certbot renew` twice daily. If a certificate has less than 30 days left, it gets renewed. After renewal, Nginx must be reloaded (`--post-hook "systemctl reload nginx"`). Monitoring expiry with a separate script is a good practice for early detection of renewal failures.

## Takeaways
1. Never rely on manual certificate renewal — automate it.
2. Certbot auto-creates renewal timers; for mkcert, create your own.
3. systemd timers are more modern and flexible than cron.
4. Always reload Nginx after certificate renewal.
5. Check certificate expiry regularly — scripts catch renewal failures early.
