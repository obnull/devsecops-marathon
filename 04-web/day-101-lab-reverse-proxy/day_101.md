# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 101: [LAB] Reverse Proxy for a Containerized App

## Objective
Independently configure Nginx as a reverse proxy with load balancing for a new application. Apply all skills from the week: upstream, proxy headers, redirects, and virtual hosts.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Application: Python 3.12, Flask 3.1 (three instances)

## Description
Deployed a new service (`api.local`) with three Flask instances behind Nginx. Wrote a custom Flask app with `/` and `/status` routes. Configured an upstream group with three servers, reverse proxy with proper headers, a `www` to non-www redirect, and separate access/error logs. Tested the full setup: round-robin distribution, redirect functionality, and log verification.

## What Was Done
- Created folder `day-101-lab-reverse-proxy` with a new `app.py`
- Wrote Flask app with two routes: `/` (HTML page) and `/status` (JSON with instance info)
- Launched three Flask instances on ports 5101, 5102, 5103
- Created Nginx config with `upstream api_backend` pointing to all three instances
- Configured `proxy_pass http://api_backend;` with `proxy_set_header` directives
- Added a separate `server {}` block for `www.api.local` → `api.local` redirect
- Set up separate logs: `api_access.log` and `api_error.log`
- Added `api.local` and `www.api.local` to `/etc/hosts`
- Tested round-robin distribution across three instances
- Tested www redirect with `curl -I` and `curl -L`

## Commands Used
- `mkdir -p day-101-lab-reverse-proxy`
- `python3 app.py 5101`
- `python3 app.py 5102`
- `python3 app.py 5103`
- `sudo tee /etc/nginx/sites-available/api.local`
- `sudo ln -s /etc/nginx/sites-available/api.local /etc/nginx/sites-enabled/`
- `sudo nginx -t`
- `sudo systemctl reload nginx`
- `echo "127.0.0.1 api.local www.api.local" | sudo tee -a /etc/hosts`
- `curl http://api.local:8000/`
- `curl http://api.local:8000/status`
- `curl -I -H "Host: www.api.local" http://127.0.0.1:8000/`
- `curl -L -H "Host: www.api.local" http://127.0.0.1:8000/`

## Verification
- `curl http://api.local:8000/` returned HTML with "API Gateway" heading
- Three consecutive `curl http://api.local:8000/status` requests returned `5101`, `5102`, `5103` — round-robin confirmed
- `curl -I` for `www.api.local` returned `301 Moved Permanently` with `Location: http://api.local:8000/`
- `curl -L` for `www.api.local` followed the redirect and returned the page
- `api_access.log` showed requests being logged correctly
- `sudo nginx -t` passed without errors

## Issues Encountered
No issues. All components worked on the first attempt.

## Key Concept
**Independent deployment.** This lab reinforced the complete reverse proxy deployment cycle: write application → launch multiple instances → define upstream → configure proxy with headers → add redirects → set up logs → test. The ability to deploy this pattern from scratch without guidance confirms understanding of Nginx as a reverse proxy and load balancer. This is a foundational pattern for microservices, API gateways, and cloud-native architectures.

## Takeaways
1. The upstream + reverse proxy pattern scales to any number of backend instances.
2. Each backend should report its identity for debugging and verification.
3. Redirects and proxy headers are essential parts of a production-grade proxy setup.
4. Separate log files per site keep debugging manageable as the number of services grows.
5. The complete deployment cycle becomes faster with repetition.
