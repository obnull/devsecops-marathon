# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 100: Redirects and Rewrite

## Objective
Learn to configure HTTP redirects and URL rewriting in Nginx. Understand the difference between `return` and `rewrite`, and when to use each.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Application: Python 3.12, Flask 3.1

## Description
Configured two types of redirects: `www` to non-www (permanent redirect) and an old URL to a new one. Used `return 301` for simple redirects and learned about the `$request_uri` variable to preserve the request path. Tested redirects with `curl -I` (show headers) and `curl -L` (follow redirects).

## What Was Done
- Added a separate `server {}` block for `www.proxy.local` that redirects to `proxy.local`
- Used `return 301 http://proxy.local:8000$request_uri;` for the www redirect
- Added `location = /old-page { return 301 /; }` inside the main server block
- Verified that `curl -I` shows `301 Moved Permanently` with the correct `Location` header
- Verified that `curl -L` follows the redirect and returns the final page
- Confirmed that the request path is preserved when redirecting from www

## Commands Used
- `sudo nano /etc/nginx/sites-available/proxy.local`
- `sudo nginx -t`
- `sudo systemctl reload nginx`
- `curl -I -H "Host: www.proxy.local" http://127.0.0.1:8000/`
- `curl -L -H "Host: www.proxy.local" http://127.0.0.1:8000/`
- `curl -I -H "Host: proxy.local" http://127.0.0.1:8000/old-page`
- `curl -L -H "Host: proxy.local" http://127.0.0.1:8000/old-page`

## Verification
- `curl -I` for `www.proxy.local` returned `301 Moved Permanently` with `Location: http://proxy.local:8000/`
- `curl -L` for `www.proxy.local` followed the redirect and returned the Flask page
- `curl -I` for `/old-page` returned `301` with `Location: /`
- `curl -L` for `/old-page` followed the redirect to the home page
- Normal requests to `proxy.local` still work and balance between Flask instances

## Key Concept
**Redirects vs rewrites.** `return 301 URL;` is the simplest and preferred way to redirect clients. It tells the browser "this resource moved permanently" and provides the new location. `rewrite` is more powerful â€” it can transform URLs using regular expressions before passing them to the backend. Use `return` for simple redirects (wwwâ†’non-www, HTTPâ†’HTTPS, oldâ†’new URL). Use `rewrite` for complex URL transformations (changing path structure, appending parameters). The variable `$request_uri` contains the original request path and query string â€” essential for preserving the full URL during redirects.

## Takeaways
1. `return 301` is the standard for permanent redirects â€” it's simple and fast.
2. Use a separate `server {}` block for www-to-non-www redirects.
3. `$request_uri` preserves the path and query string during redirects.
4. `curl -I` shows response headers â€” useful for testing redirects without following them.
5. Always include the port in redirect URLs if not using the default port 80.
