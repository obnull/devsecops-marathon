# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 103: TLS Basics

## Objective
Understand how HTTPS and TLS encryption work. Create a self-signed certificate and configure Nginx for HTTPS.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Tools: OpenSSL, curl, tcpdump

## Description
Studied the three-step TLS process: handshake, encryption, integrity. Learned what a certificate and Certificate Authority are. Created a self-signed certificate, configured Nginx on port 8443, and verified encryption with curl, tcpdump, and openssl.

## What Was Done
- Studied TLS theory (symmetric/asymmetric encryption, chain of trust)
- Created a self-signed certificate: `selfsigned.crt` + `selfsigned.key`
- Configured Nginx for HTTPS on port 8443 with HTTP-to-HTTPS redirect
- Verified HTTP traffic is readable, HTTPS traffic is encrypted (tcpdump)
- Checked cipher suite with `openssl s_client`
- Compared self-signed certificate to real website certificates (google.com, github.com)

## Commands Used
- `sudo openssl req -x509 -nodes -days 30 -newkey rsa:2048 -keyout /etc/nginx/ssl/selfsigned.key -out /etc/nginx/ssl/selfsigned.crt -subj "/CN=proxy.local"`
- `openssl s_client -connect google.com:443 -showcerts`
- `sudo tcpdump -i lo -A port 8443`
- `curl -k https://proxy.local:8443/`
- `echo | openssl s_client -connect proxy.local:8443 2>/dev/null | grep Cipher`

## Verification
- `curl http://proxy.local:8000/` returned 301 redirect to HTTPS
- `curl -k https://proxy.local:8443/` returned Flask page via HTTPS
- tcpdump showed readable text for HTTP, random bytes for HTTPS
- `openssl s_client` showed `Cipher: TLS_AES_256_GCM_SHA384`

## Issues Encountered
No issues.

## Key Concept
**TLS Handshake.** Client and server negotiate encryption: the client verifies the certificate, generates a session key, encrypts it with the server's public key. The server decrypts it with the private key. After that, all traffic uses symmetric encryption (fast). A self-signed certificate encrypts traffic but browsers do not trust it — for trust, a certificate from a Certificate Authority is required.

## Takeaways
1. HTTPS = HTTP + TLS. Without TLS, all data is plaintext.
2. A certificate binds a domain to a public key and is validated through a chain of trust.
3. Self-signed certificates are good for testing, not for production.
4. tcpdump + openssl are tools for verifying encryption in practice.
5. Port 443 is the HTTPS standard, but any port works for testing (e.g., 8443).
