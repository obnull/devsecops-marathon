# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 108: [LAB] Full HTTPS Site with Reverse Proxy

## Objective
Deploy a complete secure website from scratch: static files, Flask API, HTTPS, reverse proxy, and redirects. Apply all skills from the SSL week.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0
- Application: Python 3.12, Flask 3.1
- Certificates: mkcert

## Description
Built a production-like secure website (`secure.local`) combining static content and a Flask API behind Nginx. Configured HTTPS with a trusted mkcert certificate, HTTP-to-HTTPS redirect, www-to-non-www redirect, reverse proxy for API routes, and separate access/error logs. Tested the full setup against a checklist.

## What Was Done
- Created folder structure: `/var/www/secure.local/css/` for static files
- Deployed ready-made HTML (`index.html`) and CSS (`style.css`) — CyberShield theme
- Wrote Flask `app.py` with `/api/status` endpoint returning JSON
- Generated mkcert certificate for `secure.local` and `www.secure.local`
- Wrote Nginx config with three server blocks: www redirect, HTTP redirect, HTTPS main site
- Configured `ssl_protocols`, `ssl_ciphers`, `ssl_prefer_server_ciphers`
- Set `root` for static files and `proxy_pass` to Flask for `/api/` location
- Added `secure.local` and `www.secure.local` to `/etc/hosts`
- Launched Flask on port 5200
- Tested all checklist items with curl and browser

## Commands Used
- `sudo mkdir -p /var/www/secure.local/css`
- `sudo tee /var/www/secure.local/index.html`
- `sudo tee /var/www/secure.local/css/style.css`
- `mkdir -p ~/devsecops-marathon/04-web/day-108-lab-ssl-site`
- `python3 -m venv venv && source venv/bin/activate && pip install flask`
- `python3 app.py 5200`
- `mkcert -cert-file secure.local.pem -key-file secure.local-key.pem secure.local www.secure.local`
- `sudo tee /etc/nginx/sites-available/secure.local`
- `sudo ln -s /etc/nginx/sites-available/secure.local /etc/nginx/sites-enabled/`
- `echo "127.0.0.1 secure.local www.secure.local" | sudo tee -a /etc/hosts`
- `sudo nginx -t && sudo systemctl reload nginx`
- `curl -I http://secure.local:8080/`
- `curl -I -H "Host: www.secure.local" http://127.0.0.1:8080/`
- `curl -k https://secure.local:8443/`
- `curl -k https://secure.local:8443/api/status`

## Verification
- `curl -I http://secure.local:8080/` returned `301 Moved Permanently` with `Location: https://secure.local:8443/`
- `curl -I -H "Host: www.secure.local" http://127.0.0.1:8080/` returned `301` to `https://secure.local:8443/`
- `curl -k https://secure.local:8443/` returned HTML page (CyberShield)
- `curl -k https://secure.local:8443/api/status` returned JSON: `{"client_ip":"127.0.0.1","healthy":true,"instance":"5200","service":"CyberShield API"}`
- `sudo nginx -t` — syntax ok
- Browser showed green padlock at `https://secure.local:8443/`

## Issues Encountered
1. **curl SSL certificate problem** — curl did not trust the mkcert root CA. Not a config issue; browser trusted it. Can be fixed with `curl --cacert $(mkcert -CAROOT)/rootCA.pem`.

## Key Concept
**Complete secure site deployment.** This lab integrated all four weeks of Month 4: static site serving, reverse proxy, HTTPS with trusted certificates, and redirects. The pattern — Nginx as the single entry point handling TLS, serving static files directly, and proxying API requests to a backend — is the industry standard for web application deployment. In production, the backend would run in a container or be managed by systemd, and certificates would come from Let's Encrypt with auto-renewal.

## Takeaways
1. Nginx can serve static files and proxy API requests in a single configuration.
2. The `location /api/` block overrides the general `location /` for matching paths.
3. www and HTTP redirects should be separate server blocks for clarity.
4. A trusted certificate (mkcert or Let's Encrypt) is essential for a green padlock.
5. The full deployment cycle becomes faster with each repetition.
