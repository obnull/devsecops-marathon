# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 112: Content-Security-Policy (CSP)

## Objective
Implement a Content-Security-Policy header to prevent XSS and data injection attacks by controlling which resources the browser is allowed to load.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0

## Description
Designed and deployed a Content-Security-Policy for the secure site. Defined allowed sources for scripts, styles, images, frames, and API connections. Used `'self'` for same-origin resources, `'unsafe-inline'` for inline styles, and `data:` for inline images. Verified the policy does not break the site.

## What Was Done
- Verified no CSP header was present before the change
- Designed a CSP policy: `default-src 'self'`, `style-src 'self' 'unsafe-inline'`, `img-src 'self' data:`, `frame-src 'none'`, `connect-src 'self'`
- Added the `Content-Security-Policy` header to the server block
- Added the same header to the `/api/` location block
- Verified the header appears on responses
- Tested in browser DevTools Console — no CSP errors, site works correctly

## Commands Used
- `curl -I -k https://secure.local:8443/ 2>/dev/null | grep -i content-security`
- `sudo nano /etc/nginx/sites-available/secure.local`
- `sudo nginx -t && sudo systemctl reload nginx`
- `curl -I -k https://secure.local:8443/api/status 2>/dev/null | grep -i content-security`

## Verification
- `curl -I -k https://secure.local:8443/` returned `Content-Security-Policy: default-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; frame-src 'none'; connect-src 'self'`
- Browser DevTools Console showed no CSP-related errors
- All site functionality (CSS, API calls) worked without blocking

## Issues Encountered
No issues. The policy was designed to match existing site resources.

## Key Concept
**Content-Security-Policy as a whitelist.** CSP tells the browser which sources are trusted for each resource type: scripts, styles, images, fonts, frames, and connections. Any resource from an untrusted source is blocked. This is the most effective defense against XSS — even if an attacker injects a script tag, the browser will not execute it unless the source matches the policy. CSP requires careful design: too strict and the site breaks, too loose and it offers no protection. The recommended approach is to start strict and relax iteratively.

## Takeaways
1. CSP is the strongest defense against XSS attacks.
2. `default-src 'self'` sets the baseline for all resource types.
3. `'unsafe-inline'` allows inline styles/scripts — needed for embedded CSS but reduces security.
4. `frame-src 'none'` complements `X-Frame-Options` for clickjacking protection.
5. Always test CSP in a browser DevTools Console before deploying to production.
