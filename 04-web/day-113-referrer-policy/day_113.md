# Stage 01: Linux & Web Servers, SSL & Security Headers
## Day 113: Referrer-Policy & Permissions-Policy

## Objective
Control information leakage via the Referer header and restrict access to browser features using Referrer-Policy and Permissions-Policy headers.

## Environment
- OS: Ubuntu 24.04, Oh My Zsh
- Web Server: Nginx 1.24.0

## Description
Added `Referrer-Policy: strict-origin-when-cross-origin` to limit the information sent in the Referer header when navigating to other sites. Added `Permissions-Policy` to disable camera, microphone, and geolocation access for the site. Verified both headers appear on all responses.

## What Was Done
- Verified no Referrer-Policy or Permissions-Policy headers were present
- Added `Referrer-Policy "strict-origin-when-cross-origin" always;` to the server block
- Added `Permissions-Policy "camera=(), microphone=(), geolocation=()" always;` to the server block
- Duplicated both headers in the `/api/` location block
- Verified headers appear on static and API responses
- Confirmed all 6 security headers are now present together

## Commands Used
- `curl -I -k https://secure.local:8443/ 2>/dev/null | grep -i "referrer-policy\|permissions-policy"`
- `sudo nano /etc/nginx/sites-available/secure.local`
- `sudo nginx -t && sudo systemctl reload nginx`
- `curl -I -k https://secure.local:8443/ 2>/dev/null | grep -E "X-Frame|X-Content|Strict|Content-Security|Referrer|Permissions"`

## Verification
- `curl -I -k https://secure.local:8443/` returned `Referrer-Policy: strict-origin-when-cross-origin` and `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- All 6 security headers present in a single response
- API responses also included both headers

## Issues Encountered
No issues.

## Key Concept
**Referrer control and browser feature restrictions.** The Referer header can leak full URLs (including tokens and session data) to third-party sites when a user clicks a link. `Referrer-Policy` limits what information is sent. `strict-origin-when-cross-origin` sends only the domain (no path) for cross-origin requests, and nothing when downgrading from HTTPS to HTTP. `Permissions-Policy` (formerly Feature-Policy) controls which browser APIs a site can use — camera, microphone, geolocation, etc. Empty parentheses `()` mean the feature is denied to all origins. This reduces the attack surface: even if an XSS vulnerability exists, the attacker cannot access these features.

## Takeaways
1. `Referrer-Policy: strict-origin-when-cross-origin` is the recommended default — balances privacy and functionality.
2. `Permissions-Policy` denies browser features at the HTTP level, before any JavaScript runs.
3. Empty `()` means the feature is blocked for all origins, including the site itself.
4. Both headers complement CSP and HSTS for a layered defense.
5. 6 security headers together provide a strong baseline for any production website.
